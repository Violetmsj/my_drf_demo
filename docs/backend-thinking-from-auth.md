# 从注册登录开始，理解“后端拿到需求后到底在想什么”

这篇文档不是为了教你背概念，而是为了帮你建立一种感觉:

“我拿到一个功能时，脑子里至少知道先想哪几件事，不会整个人发懵。”

你现在这种状态其实很正常。很多前端同学刚接触后端时，都会有一种很强的迷茫感:

- 前端拿到需求，通常很快能想到页面长什么样、交互怎么走、状态怎么存。
- 但后端拿到需求，经常会一下子不知道从哪里开始，因为后端不像页面那样“看得见”。

后端更像是在搭一套“看不见的规则系统”。

你可以先记住一句特别重要的话:

**前端通常先想“页面怎么展示”，后端通常先想“数据怎么流动，规则怎么落地”。**

而你这个项目，刚好已经有了一个很适合入门的例子: 注册、登录、获取当前用户、退出登录。

我会一边结合你现在的代码，一边讲“后端开发者接到需求后的思考顺序”。

---

## 一、先把“后端到底在干嘛”翻译成人话

如果用最不专业的话来讲:

- 前端负责“把操作交给用户”
- 后端负责“决定这些操作能不能做、怎么做、做完数据放哪儿”

比如“注册”这件事:

- 前端做的事很像收银台: 把用户名、密码收上来，点提交
- 后端做的事很像柜台后面的工作人员:
  - 看看数据齐不齐
  - 看看用户名有没有重复
  - 看看密码合不合格
  - 合格就创建用户
  - 不合格就告诉前端哪里不行

再比如“登录”:

- 前端负责把用户名密码发过去
- 后端负责核对“这个人是不是他自己”
- 核对成功后，后端发一个“通行证”给前端
- 前端以后每次访问需要权限的接口，都带着这个通行证

在你这个项目里，这个“通行证”就是 `Token`。

所以你可以先把后端想成一句话:

**后端 = 收数据 + 验数据 + 改数据 + 返回结果 + 守规则**

## 二、先看你这个项目里的注册登录，整条链路是怎么跑的

你现在的认证相关接口都写在这里:

- `backend/course/urls.py`
- `backend/course/views.py`
- `backend/course/serializers.py`

对应路由是:

- `POST /course/auth/register/`
- `POST /course/auth/login/`
- `POST /course/auth/logout/`
- `GET /course/auth/me/`

前端调用这些接口的代码在:

- `frontend/src/services/auth.js`

### 1. 注册时发生了什么

你前端注册页提交表单后，会调用:

- `frontend/src/services/auth.js` 里的 `registerUser()`

它实际请求的是:

- `POST /api/course/auth/register/`

Vite 会把 `/api` 代理到 Django，所以后端实际收到的是:

- `POST /course/auth/register/`

然后后端路由把这个请求交给:

- `backend/course/urls.py` 第 15 行左右的 `RegisterView`

再进入:

- `backend/course/views.py` 第 46 到 53 行左右

这里的代码做了三件事:

1. 把前端传来的数据交给 `RegisterSerializer`
2. 调用 `is_valid()` 做校验
3. 校验通过后 `save()` 创建用户

真正的校验细节写在:

- `backend/course/serializers.py` 第 26 到 47 行左右

这里你可以把 serializer 理解成一个“门卫 + 翻译官”:

- 门卫: 检查数据能不能进来
- 翻译官: 把前端发来的 JSON，变成 Django 能处理的数据

在注册这个例子里，它做了这些事:

- 检查用户名是否重复
- 检查密码是否符合 Django 的密码规则
- 调用 `User.objects.create_user()` 创建用户

注意这里有一个容易忽略但很关键的点:

`create_user()` 不是单纯把密码原样存进数据库，它会帮你把密码处理成安全格式。  
你可以先把它理解成“不会把真实密码明文裸奔地存起来”。

### 2. 用户创建成功后，Token 是怎么来的

你在 `backend/course/views.py` 第 32 到 43 行附近写了一个信号函数 `generate_token()`。

这段代码的意思很像:

“只要系统里新创建了一个用户，我就顺手给他发一个 Token。”

也就是说:

- 前端调用注册接口
- 后端创建用户
- Django 触发 `post_save`
- 你的信号函数自动创建这个用户的 Token

这个设计的好处是:

- 注册逻辑里不用每次手写 `Token.objects.create(...)`

它的代价是:

- 对新手来说会有点“代码没写在眼前，但事情已经发生了”的魔法感

所以你学习时要特别记住:

**后端里有些事不是写在主流程里直接执行，而是靠“钩子”自动触发。**

现在这个项目里，`post_save` 就是这样一个钩子。

### 3. 登录时发生了什么

登录逻辑在:

- `backend/course/views.py` 第 56 到 70 行附近

流程是:

1. 前端把 `username` 和 `password` 发过来
2. 后端把数据交给 `LoginSerializer`
3. `LoginSerializer` 调用 Django 的 `authenticate()`
4. 如果用户名密码对得上，就拿到这个用户对象
5. 后端再取出这个用户的 Token
6. 把 `token` 和 `user` 一起返回给前端

真正做“账号密码对不对”的地方在:

- `backend/course/serializers.py` 第 50 到 69 行附近

这里你可以把 `authenticate()` 理解成:

“把用户名密码拿去问 Django: 这个人说自己是谁，是真的吗？”

如果验证失败，就返回:

- `用户名或密码错误`

如果成功，前端会把返回的 `token` 存到:

- `localStorage`

对应代码在:

- `frontend/src/services/auth.js` 第 17 到 20 行、第 109 到 118 行附近

### 4. 为什么登录后还能知道“我是谁”

登录成功后，前端会把 Token 存起来。  
之后访问需要登录的接口时，会在请求头里带上:

- `Authorization: Token xxxxx`

对应代码在:

- `frontend/src/services/auth.js` 第 71 到 79 行附近

后端为什么能认出你？

因为你在 `backend/mydrfdemo/settings.py` 里配置了:

- `TokenAuthentication`
- `SessionAuthentication`
- `BasicAuthentication`

位置大概在第 127 到 150 行附近。

这段配置可以先翻译成一句大白话:

“DRF 啊，你收到请求后，可以用这几种方式来判断用户身份。”

当请求里带着 Token 时，DRF 就会尝试把这个 Token 对应到某个用户。  
如果找到了，就会把这个用户放到 `request.user` 里。

所以在:

- `backend/course/views.py` 第 83 到 87 行的 `CurrentUserView`

你才能直接写:

- `request.user`

这很像前端里“登录态恢复以后，全局 store 里已经有当前用户了”。

### 5. 退出登录时发生了什么

退出接口在:

- `backend/course/views.py` 第 73 到 80 行附近

这段逻辑很简单:

- 如果当前请求带了 Token
- 就把这个 Token 删掉

删掉以后，这个 Token 就失效了。  
前端再拿它去访问 `me` 接口，就会变成未登录。

测试代码里也验证了这一点:

- `backend/course/tests.py` 第 72 到 82 行附近

## 三、真正重要的部分: 后端拿到一个需求，第一反应应该想什么

这部分才是你最需要建立的能力。

很多新手一拿到需求，就会立刻想:

- 我要写哪个 view？
- 我要不要建 serializer？
- 这个接口怎么返回？

这些当然要想，但它们不是第一步。

后端第一步应该先想下面这 5 个问题。

### 1. 这个功能到底在处理谁的数据

这是第一反应。

你先不要想代码，先想“这件事到底是在改什么数据”。

比如注册登录:

- 注册是在创建 `User`
- 登录本质上不是创建新数据，而是在校验用户身份，然后发一个可用凭证
- 获取当前用户，是读取当前登录用户的信息
- 退出登录，是让当前凭证失效

如果你拿到的是“课程收藏”需求，你第一反应就应该是:

- 收藏的是谁和谁的关系？
- 是 `用户收藏课程` 这条关系数据吗？

如果你拿到的是“评论”需求，你第一反应应该是:

- 评论本身是不是一张表？
- 它跟课程、用户是什么关系？

后端很大一部分工作，其实就是先把“需求描述”翻译成“数据关系”。

### 2. 这个功能的规则是什么

后端最核心的价值，不是会写接口，而是会守规则。

还是注册登录:

- 用户名不能重复
- 密码不能太弱
- 没登录的人不能查看个人信息
- 退出后旧 Token 不能继续用

课程功能里你也已经有类似规则了:

- 登录后才能访问课程接口
- 只有课程创建者才能修改课程

这就是后端思维里非常重要的一句话:

**先写规则，再写代码。代码只是规则的搬运工。**

### 3. 谁能调用这个功能，谁不能

这其实就是“权限”。

你现在项目里的全局默认权限是:

- `IsAuthenticated`

也就是:

- 默认都要先登录

这个配置在 `backend/mydrfdemo/settings.py` 第 143 到 145 行附近。

但注册和登录接口为什么不用登录也能访问？

因为它们单独写了:

- `permission_classes = (AllowAny,)`

这就是一个很典型的后端思路:

- 先有全局规则
- 再给少数特殊接口开绿灯

你以后做需求时，脑子里要习惯先问一句:

**这个接口到底是公开的，还是必须登录后才能访问？**

### 4. 成功时返回什么，失败时返回什么

前端很习惯想“接口给我什么字段”，后端也要想，但后端还要多想一层:

- 什么情况下算成功
- 什么情况下算失败
- 失败时应该告诉前端什么

比如注册:

- 成功: 返回新用户信息，状态码 `201`
- 失败: 用户名重复，返回错误信息，状态码 `400`

比如登录:

- 成功: 返回 `token + user`
- 失败: 返回“用户名或密码错误”

所以一个成熟一点的后端，不是只会“写 happy path”，还会先把失败路径想清楚。

### 5. 这个功能要拆成哪几层

拿 Django + DRF 来说，你可以先这样粗暴理解:

- `urls.py`: 门牌号，决定请求进哪个房间
- `views.py`: 接待员，决定这次请求怎么处理
- `serializers.py`: 检查和整理数据
- `models.py`: 数据长什么样
- `permissions.py`: 谁能做什么
- `tests.py`: 证明这套东西真的按预期工作

如果你现在还不熟，不要怕。  
你不用一开始就想得特别细，只要先形成这个意识:

**一个功能通常不是塞进一个文件就完事，而是会拆在几个位置共同完成。**

## 四、把“后端思考顺序”套到注册登录这个需求上

现在我们假装你什么代码都还没写，只是产品跟你说:

“做一个最简单的注册登录。”

这时候后端比较健康的思考顺序应该是这样的。

### 第一步: 先把需求翻成人话

不要急着开写，先自己用最土的话复述一遍:

- 用户可以注册一个账号
- 注册时要输入用户名和密码
- 用户名不能重复
- 密码不能太弱
- 注册成功后可以去登录
- 登录成功后后端要告诉前端“你现在是谁”
- 后续需要登录的接口，要能识别这个用户
- 用户可以退出登录

这一段看起来很废话，但其实特别重要。  
因为很多人一上来就写代码，写到一半才发现自己连“成功标准”都没想清楚。

### 第二步: 想清楚需要哪些接口

注册登录这个需求，最少要拆成这几个接口:

- 注册
- 登录
- 获取当前用户
- 退出登录

这一步很像前端先想“我这个页面至少需要哪些交互动作”。

只不过后端想的是“系统对外至少要提供哪些入口”。

### 第三步: 想清楚每个接口收什么、做什么、回什么

这是后端开发最重要的习惯之一。  
你甚至可以先不写代码，先拿纸列表。

#### 注册接口

- 收什么: `username`、`password`
- 做什么:
  - 检查用户名是否重复
  - 检查密码是否合法
  - 创建用户
- 回什么:
  - 成功返回用户信息
  - 失败返回错误原因

#### 登录接口

- 收什么: `username`、`password`
- 做什么:
  - 验证用户名密码
  - 找到该用户的 Token
- 回什么:
  - `token`
  - `user`

#### 当前用户接口

- 收什么: 请求头里的 Token
- 做什么:
  - 根据 Token 找到当前用户
- 回什么:
  - 当前用户信息

#### 退出接口

- 收什么: 当前 Token
- 做什么:
  - 删除当前 Token
- 回什么:
  - 退出成功

看到这里你应该会慢慢感受到:

**后端不是“突然开始写代码”，而是先把接口当成一组有输入、有规则、有输出的小机器。**

### 第四步: 决定数据和规则放在哪儿

在 DRF 里，一个很实用的入门思路是:

- 数据结构，放 `models.py`
- 数据校验，尽量放 `serializers.py`
- 请求流程，放 `views.py`
- 权限控制，放 `permissions.py`
- 路由映射，放 `urls.py`

所以注册登录这个需求里:

- 用户数据本身，直接复用 Django 自带的 `User`
- 注册/登录参数校验，放进 serializer
- 注册/登录流程控制，放进 view
- 哪些接口可以匿名访问，写在 view 的权限类里

这样一来，你的脑子就不会乱成一团，因为每一类事情都有固定归位的地方。

### 第五步: 再开始写代码

注意顺序:

不是“我会什么就先写什么”，而是“我已经知道要做什么，再把代码填进去”。

这就是为什么一个后端新人最先要练的，不是背 API，而是练“拆问题”。

## 五、如果以后你要独立开发一个新功能，可以按这个顺序来

你可以把下面这套流程，当成你自己的后端起手模板。

### 第 1 步: 先别写代码，先回答 6 个问题

每接到一个需求，先写下这 6 个问题的答案:

1. 这个功能在处理什么数据？
2. 需要新增表吗？还是用现有表就够了？
3. 需要几个接口？
4. 每个接口收什么参数？
5. 每个接口的规则是什么？
6. 哪些人能调这个接口？

如果这 6 个问题你答不出来，说明你还没到写代码的时候。

### 第 2 步: 画最小流程

比如“注册登录”最小流程就是:

- 注册
- 登录
- 带 Token 获取当前用户
- 退出

比如“创建课程”最小流程就是:

- 登录
- 提交课程信息
- 后端保存课程，并把当前用户设为讲师

你现在项目里在创建课程时，就是这样做的:

- `serializer.save(teacher=request.user)`

也就是:

- 前端不用传老师是谁
- 后端直接用“当前登录用户”来填

这就是很典型的后端思维:

**凡是可以由系统自己确定的东西，尽量不要让前端传。**

因为前端传的内容，用户可以改；后端自己算出来的内容，更可靠。

### 第 3 步: 先写校验，再写主流程

新手很容易先把主流程写得很热闹，最后才想“哦对，还要校验”。

但更稳的顺序是:

- 先想好哪些情况不能通过
- 再写通过时怎么处理

比如注册:

- 用户名重复，不通过
- 密码太弱，不通过
- 都通过，才创建用户

这就是为什么 serializer 在 DRF 里很重要。  
它能帮你把很多“先拦住错误数据”的工作，放到比较统一的位置。

### 第 4 步: 先跑通最小闭环，不要一口吃成胖子

拿注册登录来说，第一版你只要做到:

- 能注册
- 能登录
- 能拿到当前用户
- 能退出

就已经是一个完整闭环了。

不要一开始就给自己加太多戏，比如:

- 邮箱注册
- 手机验证码
- 找回密码
- 图形验证码
- 记住我
- 多端登录管理

这些都可以以后慢慢加。  
后端学习最怕的不是简单，而是一开始就把自己吓住。

### 第 5 步: 最后补测试

你这个项目其实已经有一个很好的习惯了:

- `backend/course/tests.py`

里面把注册、登录、获取当前用户、退出都测了一遍。

你可以把测试理解成:

“我写了一套规则，现在我得证明它真的管用。”

对后端来说，测试很重要，因为后端很多问题不是页面肉眼一看就能发现的。

比如:

- 退出后 Token 还在不在？
- 错密码会不会误登录？
- 重复用户名会不会被创建？

这些都更适合靠测试兜底。

## 六、你作为前端开发，最容易卡住的点是什么

这一段我专门按前端视角写。

### 1. 你总想先看到“页面”

这很正常，因为你熟悉的是可视化结果。  
但后端很多工作在开始阶段是“没有页面”的。

所以你可以强行给自己换个起点:

不要先问“页面怎么做”，先问:

- 我现在在处理哪张表？
- 我要提供哪个接口？
- 这个接口成功和失败分别是什么？

### 2. 你会下意识把后端当成“接口搬运工”

前端常见的感觉是:

- 后端不就是收参数、查数据库、返回 JSON 吗？

但后端真正麻烦的地方在于:

- 规则放哪儿
- 权限怎么管
- 数据谁来兜底
- 错误怎么处理
- 以后好不好扩展

也就是说:

**后端最难的不是“写通”，而是“写得稳”。**

### 3. 你容易看到代码，却看不到“责任分层”

前端写久了，可能会觉得:

- 这个校验写 view 里也能跑
- 这个权限写 if 里也能跑
- 这个返回值随手拼一下也能跑

确实能跑。

但后端慢慢会更在意:

- 哪部分是数据校验
- 哪部分是请求流程
- 哪部分是权限规则
- 哪部分是数据库结构

因为一旦功能变多，不分层就会越来越乱。

所以你现阶段不用追求“写得多高级”，但要开始建立“不同问题放不同地方”的意识。

## 七、把你当前项目当成一个学习地图

你这个项目其实挺适合学习，因为它不是特别大，而且认证功能和课程功能都已经串起来了。

你可以按这个顺序去啃:

### 第一阶段: 先只看注册登录

建议按这个顺序读:

1. `frontend/src/views/RegisterView.vue`
2. `frontend/src/services/auth.js` 里的 `registerUser()`
3. `backend/course/urls.py`
4. `backend/course/views.py` 里的 `RegisterView`
5. `backend/course/serializers.py` 里的 `RegisterSerializer`
6. `backend/course/tests.py` 里的注册测试

目标不是背代码，而是回答一句话:

“一个注册请求，是怎么从前端一路走到数据库的？”

### 第二阶段: 再看登录和登录态恢复

建议按这个顺序读:

1. `frontend/src/views/LoginView.vue`
2. `frontend/src/services/auth.js` 里的 `loginUser()`、`fetchCurrentUser()`、`restoreSession()`
3. `backend/course/views.py` 里的 `LoginView`、`CurrentUserView`、`LogoutView`
4. `backend/mydrfdemo/settings.py` 里的认证配置
5. `frontend/src/router/index.js` 里的路由守卫

目标是回答:

“为什么页面刷新后，系统还能知道我登录了没有？”

### 第三阶段: 再看课程接口

这时候你去看课程的增删改查，会比一开始轻松很多。  
因为课程接口和认证接口本质上都在做同一件事:

- 收请求
- 校验数据
- 查或改数据库
- 做权限判断
- 返回结果

只是课程功能多了一层“对象归属关系”。

比如:

- 创建课程时，把当前用户记成 `teacher`
- 更新课程时，只有这个 `teacher` 才能改

这就是从“登录态”自然过渡到“业务权限”。

## 八、你以后开发功能时，可以直接照着问自己的问题清单

以后你每写一个后端功能，都先问自己下面这些问题。

### 功能理解

- 这个功能到底要解决什么问题？
- 它操作的核心数据是什么？
- 是新增数据、修改数据、查询数据，还是删除数据？

### 接口设计

- 需要几个接口？
- 每个接口的请求方法是什么？`GET`、`POST`、`PUT`、`DELETE`？
- 每个接口收什么参数？
- 每个接口成功返回什么？
- 每个接口失败返回什么？

### 权限和规则

- 这个接口要不要登录？
- 谁都能操作，还是只有本人能操作？
- 哪些字段必须校验？
- 哪些数据不能相信前端，必须后端自己决定？

### 落地位置

- 路由写在哪？
- 视图写在哪？
- 校验写在哪？
- 权限写在哪？
- 测试写了吗？

只要你能把这些问题答出来，你就已经不是“完全懵”的状态了。

## 九、送你一个最实用的心法

你现在不用逼自己一上来就像一个老后端那样思考。  
你只需要先练一个动作:

**每拿到一个需求，先别写代码，先写“数据、规则、权限、输入、输出”这 5 件事。**

你会发现，一旦这 5 件事写出来，代码反而只是最后的体力活。

如果你什么都不写，直接冲去写代码，就会特别容易进入这种状态:

- 写了一个 view
- 写着写着发现忘了校验
- 补校验时发现权限没想好
- 改权限时又发现返回值乱了
- 最后整个人越来越晕

这不是你笨，这是因为你少了“先拆问题”这一步。

## 十、给你一个适合现在的练习方式

不要一下子学太多。  
你可以就围绕这个项目，按下面的节奏练。

### 练习 1

不看代码，自己写出“注册接口”的处理流程，哪怕只写中文也行。

比如:

- 收到用户名和密码
- 检查用户名是否重复
- 检查密码是否合法
- 创建用户
- 返回用户信息

### 练习 2

对着代码，找到每一步分别落在哪个文件。

### 练习 3

自己尝试加一个非常小的功能，比如:

- 注册时要求用户名长度至少 3 位
- 或者 `me` 接口多返回一个字段

这个练习的目的不是功能本身，而是让你体会:

- 一个小需求从“想”到“改”会经过哪些位置

### 练习 4

自己先写“我要改哪些文件，为什么改”，然后再动手。

这个习惯会非常有帮助。  
因为它本质上是在训练你的后端拆解能力。

## 最后一句

你现在最缺的，不是再多背几个 Django 名词，而是建立一条稳定的思考路线。

从你这个项目出发，最值得反复练的不是“怎么把代码写出来”，而是:

**拿到一个功能后，我能不能先把它拆成数据、规则、权限、接口、测试。**

只要这条路线慢慢长出来，你就会从“照葫芦画瓢”，一点点走到“我自己也能把葫芦画出来”。
